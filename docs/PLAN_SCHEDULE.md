# System-Arti：Hyperliquid 演算法交易平台 — 一個月計畫

> 目標：在約 **4 週**內，從「可連線、可看資料」進展到「可回測、可受控地接觸實盤」的平台雛形。  
> 技術棧：**Next.js + TypeScript（前端）**、**Python（資料、策略、下單）**。

---

## 第 1 週：基礎連線與儀表板骨架

| 天數 | 工作項目 | 產出／驗收 |
|------|----------|------------|
| 1–2 | 確認本機環境：`web` 與 `python` venv；文件化啟動指令（前端、後端 API）。 | 一鍵或兩步可同時看到前端與 Python API `health`。 |
| 3–4 | Python：`hyperliquid-python-sdk` 唯讀流程（`meta`、`all_mids`、可選 `user_state`）；錯誤處理與逾時。 | API 回傳穩定 JSON；無私鑰時不暴露敏感操作。 |
| 5 | Next：儀表板版面（帳戶摘要、市場列表占位、連線狀態）。 | 從後端代理讀取狀態並顯示（輪詢即可）。 |
| 6–7 | 設定：`.env` 分離（前端公開 URL vs 後端密鑰）；釐清主網／測試網。 | 範例 env 與 README 片段一致；本週小結與下週待辦。 |

**本週里程碑：** 能在畫面上確認「連得上 Hyperliquid、看得到公開／唯讀資料」。

---

## 第 2 週：即時資料與資料流

| 天數 | 工作項目 | 產出／驗收 |
|------|----------|------------|
| 1–2 | WebSocket：訂閱行情或成交（依 SDK 能力選頻道）；Python 端轉發或前端直連（二選一並寫清理由）。 | 儀表板有即時價格或更新時間戳。 |
| 3–4 | 歷史／快照：拉取或快取 K 線或 tick 聚合所需欄位；時間對齊（UTC）。 | 至少一個圖表占位（可用靜態資料先接通資料管線）。 |
| 5 | 儲存（可選）：SQLite 或 Parquet 存歷史片段，方便重播與除錯。 | 能從本機檔案或 DB 讀回一段歷史給回測用。 |
| 6–7 | 監控：簡單日誌結構、請求失敗重試；前端顯示「資料延遲」。 | 人為斷網時行為可預期、可觀測。 |

**本週里程碑：** 「即時 + 可查歷史」路徑打通，為回測與策略餵資料。

---

## 第 3 週：策略介面、回測與風控雛形

| 天數 | 工作項目 | 產出／驗收 |
|------|----------|------------|
| 1–2 | 定義策略介面（訊號輸入／輸出、參數 schema）；一個 **紙上策略** 範例（不實盤）。 | 程式可單元測試策略函式（mock 行情）。 |
| 3–4 | 回測迴圈：歷史 K 線 → 策略 → 模擬倉位／PnL 曲線（簡化撮合即可）。 | 產出基本績效指標（總收益、最大回撤占位）。 |
| 5 | 風控：單筆下單上限、日內虧損停損、開關「允許實盤」旗標（預設關）。 | 設定檔或環境變數可控制；前端顯示目前模式。 |
| 6–7 | 文件：策略如何註冊、如何跑回測 CLI 或 API。 | 另一位開發者能照文件跑通最小範例。 |

**本週里程碑：** 同一套策略邏輯能跑在「歷史資料」上並產出可解讀結果。

## 第 4 週：實盤銜接、安全與上線準備

| 天數 | 工作項目 | 產出／驗收 |
|------|----------|------------|
| 1–2 | 下單路徑：僅在後端使用私鑰；簽名與 nonce／限速查官方與 SDK 建議；先 **小額／測試網**。 | 測試網或極小倉位下單成功且可稽核。 |
| 3 | 前端：下單前二次確認、顯示預估影響；緊急「全平／暫停策略」占位。 | 誤觸風險可控。 |
| 4–5 | 部署選型：前端（Vercel 等）與後端（VPS／容器）分離；秘密管理（不進 git）。 | staging 環境可演示。 |
| 6–7 | 驗收清單：備份、金鑰輪替流程、已知限制列表；下階段 backlog。 | 可給自己或協作者的「上線前檢查表」。 |

**本週里程碑：** 在測試網或小額實盤下，**完整路徑可跑通一次**，且金鑰與權限邊界清楚。

---

## 持續建議（貫穿四週）

- **安全**：私鑰只留在 Python 執行環境；前端永遠不存私鑰。  
- **漸進**：先唯讀 → 再模擬／回測 → 最後才實盤；每階段有明確「開關」。  
- **紀錄**：重大參數與版本號寫進日誌或 DB，便於事後還原決策。  
- **節奏**：每週五留 1–2 小時做回顧與調整下週優先級，避免範圍膨脹。
